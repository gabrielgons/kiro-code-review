# Report Template & Format Specification

## Overview

This document defines the exact format and structure for the code review report `.md` file generated by this power.

---

## File Naming Convention

```
code-review-{owner}-{repo}-pr-{number}-{timestamp}.md
```

Example: `code-review-gabrielgons-kiro-quest-pr-1-20250525.md`

---

## Severity Legend

The report uses a three-tier severity system with color coding:

| Emoji | Color | Level | Meaning | Action |
|-------|-------|-------|---------|--------|
| :red_circle: | Red | Critical | Security vulnerabilities, critical bugs, data loss risks, blocking issues | **Must fix before merge** |
| :orange_circle: | Orange | Medium | Potential bugs, bad practices, performance issues, maintainability concerns | **Should be addressed** |
| :green_circle: | Green | Low | Style suggestions, minor optimizations, alternative approaches, nice-to-haves | **Optional improvement** |

## GitHub Comment Format

When posting inline comments on GitHub, use this format:

### For Critical Issues (Red)

```markdown
:red_circle: **[{Category}] {Title}**

{Problem description - 2-3 sentences max}

**Suggested fix:**
```{language}
{code_suggestion}
```

> :warning: **Impact:** {Brief impact statement}
```

### For Medium Issues (Orange)

```markdown
:orange_circle: **[{Category}] {Title}**

{Problem description - 2-3 sentences max}

**Consider:**
```{language}
{code_suggestion}
```
```

### For Suggestions (Green)

```markdown
:green_circle: **[{Category}] {Title}**

{Suggestion - 1-2 sentences}

```{language}
{example_code}
```
```

### PR-Level Summary Comment

```markdown
## :mag: Code Review Summary

| Severity | Count |
|----------|-------|
| :red_circle: Critical | {count} |
| :orange_circle: Medium | {count} |
| :green_circle: Suggestion | {count} |

### Key Observations

{2-3 bullet points with the most important findings}

### Recommendation

{approve / request changes / needs discussion}

---
*Automated review by [Code Review Power](https://github.com/gabrielgons/kiro-code-review)*
```

---

## Notes on Format

1. **Code snippets must include context** - Show enough surrounding code for understanding
2. **Line numbers must be accurate** - Reference the actual diff line numbers
3. **Language detection** - Use the correct language identifier for syntax highlighting
4. **Links** - Include clickable links to files and lines when possible
5. **Emoji usage** - Keep to severity indicators only, don't overuse
6. **Conciseness** - Each finding should be understandable in 30 seconds or less
